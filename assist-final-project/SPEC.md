# Техническое задание (MVP)

**Версия:** 2.1  
**Статус:** финальный компактный вариант  
**Проект:** TeleFlow

## 1) Цель

Один локальный `userbot`:
- по команде собирает дайджест из Telegram-папки;
- принимает вручную отобранные пользователем посты в ЛС;
- сохраняет их по темам;
- по команде `/write <тема>` генерирует авторский пост только из материалов этой темы.

## 2) Границы MVP

- Только один компонент: `Telethon userbot` (без Bot API-бота).
- Локальное хранение в файлах (без PostgreSQL и без веб-интерфейса).
- Текстовые посты only (медиа без текста пропускаются).
- Язык генерации поста: русский.
- Голосовые и Whisper — не входят в MVP.

## 3) Основной сценарий

1. Пользователь вызывает `/digest`.
2. Userbot формирует дайджест из источников (на базе `28_tg_project`) и отправляет его в ЛС.
3. Пользователь читает дайджест и пересылает выбранный пост в ЛС userbot.
4. Userbot запрашивает тему.
5. Пользователь отправляет тему.
6. Userbot сохраняет запись в `data/knowledge/<тема>/entry_<id>.json`.
7. Пользователь вызывает `/write <тема>`.
8. Userbot читает только эту тему, вызывает LLM и отправляет результат в ЛС.

## 4) Команды и UX

- `forward` поста в ЛС userbot → запрос темы.
- ответ темой (или номером из списка) → сохранение в базу.
- `/digest` → собрать и отправить дайджест сейчас.
- `/write <тема>` → генерация поста.

Примечание: у userbot нет bot-inline-кнопок как у Bot API, поэтому выбор темы — текстом/номером (FSM).

## 5) Данные и структура

```text
data/
├── knowledge/
│   ├── спорт/
│   │   └── entry_<id>.json
│   ├── философия/
│   └── ...
└── drafts/                  # опционально
    ├── спорт/
    └── ...
```

`KnowledgeEntry`:

```json
{
  "id": "uuid|timestamp",
  "text": "...",
  "theme": "спорт",
  "created_at": "ISO-8601",
  "source_link": "https://t.me/..."
}
```

## 6) Жёсткие правила логики

- `/write <тема>` читает только `data/knowledge/<тема>/`.
- Если тема пуста/не найдена — ответ без вызова LLM.
- LLM не классифицирует темы и не ищет посты; только синтезирует текст из переданного контекста.
- Обработка ЛС и дайджеста работает в одном процессе userbot.
- Дайджест готовится по команде `/digest` (без обязательного автопланировщика в MVP).

## 7) Режимы запуска

- `--daemon`: основной режим, userbot всегда в ожидании и обрабатывает ЛС/FSM + команды.
- `--once`: технический режим для теста/отладки одного цикла дайджеста.

## 8) Конфиг (.env) — минимум

- `ALLOWED_USER_IDS` — кто может писать в ЛС и вызывать команды.
- `THEMES` — допустимые темы.
- `THEME_DIRS` (опц.) — маппинг тема → slug папки.
- `DATA_DIR` — корень данных (`data`).
- `WRITE_MAX_FILES` — лимит файлов на `/write`.
- `WRITE_MAX_TOTAL_CHARS` — лимит контекста.
- `OUTPUT_LANG=ru`.

Плюс переменные из `.env`: Telegram, `OPENROUTER_*` для LLM, lookback и т.д. (см. `README.md`).

## 9) Критерии готовности

- [ ] Дайджест приходит в ЛС.
- [ ] Команда `/digest` запускает сбор и отправляет дайджест в ЛС.
- [ ] Forward поста в ЛС → запрос темы → запись в `data/knowledge/<тема>/`.
- [ ] `/write <тема>` генерирует пост из файлов только этой темы.
- [ ] Пустая тема корректно обрабатывается без LLM.
- [ ] Не-whitelist пользователь не может сохранять записи/вызывать `/write`.

## 10) База для реализации

Использовать `28_tg_project` как основу:
- оставить сбор из папки и дайджест, запуск дайджеста перевести на команду `/digest`;
- добавить обработчики ЛС (FSM выбора темы);
- добавить запись `KnowledgeEntry` в папки тем;
- добавить `/write <тема>` с отдельным промптом синтеза.
