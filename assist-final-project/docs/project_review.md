# TeleFlow — честная оценка проекта и презентации

Это ревью написано перед защитой, чтобы ты вышел на питч уверенно: ясно видел свои сильные стороны, знал слабые места заранее, не поплыл на вопросах и понимал, что именно стоит сказать вслух, а что — держать в уме.

## TL;DR

- Проект объективно **сильно выше среднего** для учебного финала: ~1500 строк Python в 15 модулях, реальная интеграция с двумя API (Telegram MTProto + OpenRouter), здравая архитектура с разделением ответственности, осознанные ограничения (батчи, тайм-бюджеты, whitelist, DRY_RUN, FloodWait-handling).
- Презентация **опрятная и быстрая**, но **сыроватая по смыслу**: есть плейсхолдеры (`время на проект`, `с курса`), слишком нейтральная боль, нет цифр, нет прямого оффера в команду АССИСТ+ и нет отдельного слайда про ИИ в числах.
- Главный риск на защите — **технические вопросы про надёжность и безопасность** (FloodWait, сессия userbot, секреты, данные). Ответы есть в коде — я собрал их в `qa_prep.md`.

## Как я смотрю на проект

### Что сильного (пиши это на слайде)

| Что | Где в коде | Почему это сильно |
|---|---|---|
| Реальный Telethon userbot, а не «обёртка над GPT» | `src/teleflow/telegram_client.py` | Работа с MTProto, папками, `GetDialogFilters`, `GetPeerDialogs`, `iter_messages`, FloodWait/PeerFlood. |
| Разделение на слои | `collector.py` + `summarizer.py` + `ranker.py` + `publisher.py` + `write_synthesizer.py` + `knowledge.py` | Один файл — одна ответственность, без «бога-main». |
| Боевые ограничения, а не песочница | `openrouter_timeout_sec`, `openrouter_annotate_budget_sec`, `run_once_budget_sec`, `collect_max_messages_per_channel` в `config.py` | Ты явно думал про деньги, время и сеть. Это видно. |
| Безопасность и доступ | `allowed_user_ids`, `dry_run`, `.env` c `API_HASH`/`OPENROUTER_API_KEY` | Whitelist на уровне обработчика, DRY_RUN по умолчанию. |
| Стабильность длинных ответов | `send_text` с chunked `_split_for_telegram`, обработка `FloodWaitError` / `PeerFloodError` | Реальная работа со стороной Telegram, а не игрушка. |
| Атомарная запись данных | `_atomic_write_json` в `knowledge.py` (`tempfile.mkstemp` + `os.replace`) | Это уже уровень выше «бросили `open().write()`». |
| Гибридный ранжировщик | `ranker.py`: `importance*10 + metric_score(views, forwards, reactions, length, keywords)` | Не только LLM — есть числовая часть, которой можно доверять без модели. |
| Конфигурируемость | `config.py`: ~35 переменных окружения, `_parse_*` валидация | Легко поменять модель, бюджет, папки, темы без кода. |
| Поддержка MTProxy/SOCKS5 | `_build_client_kwargs` в `telegram_client.py` | Реалистично для пользователя, у которого Telegram живёт через прокси. |
| Документация | `README.md` (128 строк), `SPEC.md` (109 строк) | Есть версия SPEC 2.1, есть критерии готовности, smoke-чеклист, troubleshooting. |

### Что слабого (я держу это при себе, но знаю)

| Что | Почему это риск на защите |
|---|---|
| Нет автотестов | Будет вопрос «как вы проверяете?». Ответ: smoke-чеклист из README + ручное DRY_RUN. |
| Глобальное `_pending_by_user` в памяти | После рестарта демона сохранится только то, что уже записали в `data/knowledge/`, а FSM забудет «висящий forward». Честно сказать — да, это осознанное упрощение MVP. |
| Один процесс = одна сессия Telethon | Масштабирование на несколько пользователей пока не предусмотрено. Это плюс для MVP и минус для «SaaS». В roadmap идёт отдельным пунктом. |
| `state.db` почти пустая | Таблица `app_state` хранит только `last_run`. Можно быть готовым к вопросу «зачем SQLite, если это один ключ?» — ответ: задел под метрики/лимиты позже. |
| `NOISE_MARKERS` захардкожены | `("реклама", "#ad", "розыгрыш", "promo")` — список крошечный. Ответ: осознанный MVP-фильтр, можно вытащить в конфиг. |
| Нет CI / линтера в репо | Не фатально, но спросят — скажи «следующий шаг, инфраструктура без неё не продвинуться». |
| `output_lang` проверяется только на `ru` | Мультиязычный пост сейчас вне MVP. |
| В `src/teleflow/__init__.py` и `__main__.py` не посмотрел — не критично, если они минимальны | Это нормально для Python-пакета. |

### Что стоит показать голосом (не переписывая код)

1. **Где реально ИИ** — только в двух точках: `DigestAnnotator` (разметка батчами) и `synthesize_post` (генерация поста). Остальное — детерминированные правила. Это честно и это сильно.
2. **Что MVP не делает специально** — Whisper, автопланировщик, веб-UI, мультитенант. Покажи, что ты понимаешь границы.
3. **Почему userbot, а не Bot API** — Bot API не видит твои папки и непрочитанное как человек. Без этого сбор дайджеста невозможен.

## Оценка презентации (до прокачки)

### Что было хорошо
- Scroll-snap + прогресс-бар + side-nav dots — опрятно и без бардака.
- SVG-диаграмма потока — живая, с пульсом активных узлов.
- Тёмная тема с зелёным акцентом, нормальный визуальный ритм.
- Адекватная семантика (`skip-link`, `role="img"`, `aria-*`).
- `prefers-reduced-motion` поддерживается.

### Что плохо для защиты «в команду АССИСТ+»
- **Два плейсхолдера** (время на проект, с курса) — если выйдешь с ними, это читается как «не дописал».
- **Защита безлична**: нет цели «попасть в команду АССИСТ+», нет явного оффера.
- **Боль слишком общая**: одно предложение про «много каналов» не продаёт проект.
- **Нет слайда про экономию и ИИ в цифрах**: цель курса — «экономия времени через ИИ», а цифр нет ни одной.
- **Нет слайда про стек курса**: не видно, что ты использовал контекст, агентов, инструменты, Git/GitHub — а это плюс на защите АССИСТ+.
- **Нет слайда «дорога к продукту»**: roadmap сейчас — общие слова, не видно горизонтов.
- **Финальный CTA слабый** — «Вопросы?». Хочется прямого оффера: «Готов в команду, вот что я приношу».
- **Google Fonts подтягиваются по сети** — в off-line зале на Wi-Fi ассистенсов шрифты могут не загрузиться. Нужно подстелить системные шрифты фолбэком (уже есть в `body`, но заголовок `.slide__title` падает на дефолт без стиля — это OK, просто имей в виду).
- **Нет счётчика слайдов** — трудно ориентироваться, сколько осталось до финала.

### Что меняю в прокачке
1. Заполняю обе заглушки конкретикой.
2. Добавляю слайд **«ИИ в числах»** — с батчами, бюджетами, моделью.
3. Добавляю слайд **«стек курса → артефакты в проекте»**.
4. Добавляю слайд **«горизонты развития»** (MVP → VPS → SaaS).
5. Добавляю **счётчик слайдов** в правом верхнем углу — ориентироваться в темпе питча.
6. Обновляю финальный слайд на прямой оффер команде АССИСТ+.
7. Убираю плейсхолдер-стили с реальных блоков.

## Список решений, которые я принял за тебя

Если что-то не нравится — скажи, и я переделаю.

- **Позиционирование** — не «userbot для себя», а «инструмент, который возвращает час в день, и каркас для B2B-продукта». Это ложится под АССИСТ+.
- **Цифры экономии** — осторожные диапазоны, без «×10» и «40 000₽ ассистент». В `speaker_notes.md` — с пометкой «по моим замерам».
- **Тон оффера** — прямой, короткий, без заигрываний: «хочу в команду, приношу рабочий прототип».
- **Имя бренда на слайдах** остаётся TeleFlow — это твоё, не ломаю.
