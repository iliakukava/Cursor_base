# Q&A prep — защита TeleFlow

Регламент — 5 минут на вопросы. Ниже 25 реальных технических вопросов, сгруппированных по темам, с короткими опорами. Отвечай спокойно, 2–4 предложения: сначала прямой ответ, потом обоснование или ссылка на код.

Все ответы опираются на то, что реально написано в `src/teleflow/*`. Если ревьюер задаст вопрос, на который нет прямого ответа — не выдумывай, честно скажи «в MVP это упрощено, вот куда бы довёл дальше».

---

## 1. Продукт и границы

**Q1. Что такое TeleFlow одним предложением?**
Локальный Telegram userbot на Telethon, который собирает дайджест непрочитанного из выбранных папок с LLM-разметкой через OpenRouter, сохраняет посты по темам в локальной базе и по команде генерирует авторский пост только из этих материалов.

**Q2. Кто целевой пользователь?**
Автор Telegram-канала, эксперт или контент-менеджер, у которого десятки каналов в подписке и одна-две темы собственного производства. В B2B-перспективе — клиенты АССИСТ+, которым нужен такой ассистент как сервис.

**Q3. Почему userbot, а не Bot API?**
Bot API не видит мои папки и непрочитанное как человек — а без этого дайджест собрать нельзя. Userbot — единственный способ честно работать с `GetDialogFiltersRequest` и `GetPeerDialogsRequest`. Для B2B-версии логично перейти на Bot API именно потому, что там продавать «один бот — много клиентов».

**Q4. Что не входит в MVP и почему?**
Whisper-голосовые, автопланировщик, веб-UI, мультитенант. Осознанное решение — сначала собрать сильное ядро «дайджест + база + синтез», потом обвязка. Это прописано в `SPEC.md` раздел 2.

## 2. Архитектура и код

**Q5. Как разложена архитектура?**
Один процесс, 16 модулей с одной ответственностью: `telegram_client` — работа с MTProto, `collector` — сбор кандидатов с дедупом по fingerprint, `summarizer` — LLM-разметка батчами, `ranker` — числовой рейтинг, `publisher` — сборка текста дайджеста, `handlers` — команды ЛС и FSM, `knowledge` — чтение/запись базы тем + микро-CRUD, `fingerprint` — SHA-256 нормализованного текста, `write_synthesizer` — синтез поста со стилями, `openrouter` — HTTP-клиент на stdlib urllib, `config` — парсинг .env. Плюс `main`, `models`, `fsm`, `state_store`, `digest_service`, `filters`.

**Q6. Почему OpenRouter, а не прямой OpenAI SDK?**
OpenRouter — единый HTTP API к десяткам моделей: смена модели через `OPENROUTER_MODEL` в .env без правок кода. Дополнительно я сознательно не тащу SDK — вызов сделан на stdlib `urllib.request`, ни одной лишней зависимости. Это видно в `openrouter.py`.

**Q7. Почему нет тестов?**
Честно: в MVP сделана ручная верификация через smoke-чеклист в `README.md` и DRY_RUN режим. Следующий шаг — покрыть `fingerprint`, `ranker`, `filters`, `knowledge` юнит-тестами (они чистые, мок не нужен), а Telethon и OpenRouter — контрактными моками. Это я понимаю как технический долг №1.

**Q8. Как устроен FSM для выбора темы и undo?**
Два слоя. FSM выбора темы — в `fsm.py`, in-memory dict `_pending_by_user`: при forward кладётся, при ответе с темой забирается и чистится. Это эфемерное состояние — если пересылка «зависла» без ответа темой, после рестарта она забывается, и это ок. А вот `/undo` — другое: путь последнего сохранённого файла хранится уже в `state.db`, per user, с ключом `last_knowledge_entry:{user_id}`. То есть откат переживает рестарт демона. В проде добавлю TTL.

**Q9. Зачем SQLite, если там было одну строку?**
Задел оказался обоснованным. Сейчас в `state.db` — `last_run` плюс per-user `last_knowledge_entry` для `/undo`. Я реализовал поверх `app_state(key, value)` обобщённые `_set_value/_get_value/_delete_value`. Следующий логичный шаг — таблица `digest_runs` с метриками запусков и индекс `fingerprint → path` для O(1) дедупа.

**Q10. Как работает ранжирование?**
Гибридное. В `ranker.py` финальный скор = `importance * 10 + metric_score`. `importance` — от LLM (1-10), `metric_score` — детерминированная формула от `log1p` views/forwards/reactions + длина + совпадения ключевых слов. Это важно: если модель сломалась, порядок всё равно не развалится.

## 3. ИИ и OpenRouter

**Q11. Как именно модель размечает посты?**
В `summarizer.py` — батчи по `OPENROUTER_DIGEST_BATCH_SIZE` (по умолчанию 8) с обрезкой текста поста до `OPENROUTER_DIGEST_TEXT_CHARS` (1600). Системный промпт требует строго JSON-массив с полями `idx, keep, importance, category, summary`. Если модель сорвала JSON — есть парсер-fallback в `_parse_json`, который пытается вытянуть `[...]` из ответа; если и это не вышло — подставляется дефолтный summary из первых 280 символов самого поста.

**Q12. Что если модель выдаёт плохой JSON или галлюцинирует?**
Три уровня защиты. Первое — `temperature=0.2` на разметке. Второе — парсер терпимый, ловит JSON внутри мусора. Третье — детерминированный метрический скор в ranker всё равно отработает, даже если LLM-часть сбойнет на всём батче. То есть дайджест выйдет, просто без умных summary.

**Q13. Как контролируется стоимость?**
Тремя параметрами из конфига: `OPENROUTER_ANNOTATE_BUDGET_SEC` (300 с — общий таймер), `OPENROUTER_DIGEST_MAX_ITEMS` (80 постов максимум уходят в LLM) и `OPENROUTER_DIGEST_TEXT_CHARS` (обрезка входа). Плюс дефолтная модель — `openai/gpt-4o-mini`, самая дешёвая адекватная. Пример: 80 постов по 1600 символов в батчах по 8 = ~20 запросов, каждый ≤ 8192 max_tokens выхода. На практике — десятки рублей за дайджест.

**Q14. Почему `synthesize_post` не классифицирует темы?**
В `SPEC.md` раздел 6 это жёсткое правило: LLM не ищет посты и не классифицирует темы — она только синтезирует текст из переданного контекста. Классификация — ручная (пользователь выбирает тему при forward). Это гарантирует, что `/write спорт` действительно читает только `data/knowledge/спорт/`, и никакая модель не притащит в пост чужую тему.

**Q15. Как работает `--style` у `/write` и зачем он нужен?**
Это карта «ключ → overlay-промпт», которая грузится из `.env` (либо `WRITE_STYLES_JSON`, либо путь к файлу `WRITE_STYLES_PATH`). Например: `{"короткий": "Пиши не длиннее 600 знаков, одна мысль", "пост-вопрос": "Закончи открытым вопросом к аудитории"}`. На `/write спорт --style=короткий` overlay просто подмешивается в системный промпт к базовым правилам («синтезируй, не пересказывай, не выдумывай»). Поменять стили = отредактировать `.env` + рестарт — не трогая код.

## 3.1. Дедупликация и база знаний

**Q16. Как устроен дедуп и зачем он?**
Модуль `fingerprint.py` — чистая функция: SHA-256 от нормализованного текста (lower, без whitespace, без эмодзи и модификаторов, обрезка до `DEDUPE_TEXT_MAX_CHARS=6000`). Используется в двух местах: в `collector.dedupe_candidates_by_fingerprint` — чтобы один и тот же пост не попал в дайджест трижды, если он перефорварден в трёх каналах. И в `knowledge.save_knowledge_entry` — перед записью сверяется по всем темам базы; если такой текст уже лежит в теме X, пользователь получает «Уже сохранено в теме X. ID: abc…». Fingerprint хранится в самом JSON-entry — при повторных проверках не нужно нормализовать заново.

**Q17. А если у пользователя уже лежат старые записи без поля `fingerprint`?**
Код это переживает: `_entry_fingerprint` в `knowledge.py` смотрит в payload и, если поля нет — пересчитывает на лету. То есть миграция не нужна, старые записи работают с новой логикой дедупа без ломки. Новые записи сохраняются уже с полем.

**Q18. Зачем команды `/list`, `/rm`, `/mv`, `/undo` — не раздуто ли?**
Это минимальный CRUD поверх базы знаний, без которого база — tar pit: накопить можно, разобрать нельзя. `/list` даёт превью, `/rm` удаляет ошибки, `/mv` чинит неверно выбранную тему, `/undo` — мгновенный откат «последнее нажатие». ID — это 32 hex-символа, regex `^[a-f0-9]{32}$` защищает от path-traversal, а `/undo` дополнительно проверяет, что целевой путь лежит внутри `data/knowledge/` через `Path.resolve().parents`.

## 4. Надёжность и безопасность

**Q19. Что с FloodWait и лимитами Telegram?**
В `telegram_client.py` у `send_text` обработан `FloodWaitError` — ждём `exc.seconds + 2` и ретраим кусок. `PeerFloodError` — логируем и пробрасываем наверх, в handler юзеру падает человеческое сообщение с подсказкой включить `DRY_RUN=false` и отправку в канал. Между частями длинного дайджеста — пауза 0.55 с.

**Q20. Как устроена безопасность?**
Четыре слоя. Whitelist `ALLOWED_USER_IDS` — первая проверка в `on_new_message`. Секреты — только в `.env`, не в коде, `.env` в `.gitignore`. Сессия Telethon — в локальном файле, неинтерактивный запуск без сессии падает с явной ошибкой, а не с непонятным фризом. `DRY_RUN=true` по умолчанию — никакой случайной отправки в канал.

**Q21. Где хранятся данные и что с приватностью?**
Всё локально: `data/knowledge/<тема>/entry_<uuid>.json` и `state.db`. Запись — атомарная через `tempfile.mkstemp` + `os.replace`, каждая запись содержит `fingerprint` для дедупа. В OpenRouter уходит только текст постов, которые пользователь сам переслал или которые пришли из его же папок. Никаких телефонов, user_id или контактов в LLM не уходит.

**Q22. Почему SOCKS5 и MTProxy?**
Потому что в России Telegram без прокси — это билет в «timeout». `_build_client_kwargs` в `telegram_client.py` поддерживает оба варианта: SOCKS5 (через python-socks) и MTProxy (через `ConnectionTcpMTProxyRandomizedIntermediate`). Плюс раздельные таймауты на connect (`TG_CONNECT_TOTAL_SEC`) и RPC (`TG_RPC_TIMEOUT_SEC`) — чтобы диагностировать «виснет на TCP» vs «виснет на авторизации».

## 5. Развитие и оффер

**Q23. Как масштабировать на много пользователей?**
Сейчас — не масштабируется, и это правда: один процесс = одна Telethon-сессия = один автор. B2B-версия — это переход на Bot API (он мультитенантен из коробки), вынос данных из файлов в Postgres, добавление биллинга и per-tenant лимитов на токены OpenRouter. Архитектура к этому готова: знание о теме и пользователе уже разделены, а `state.db` уже используется per-user (ключи вида `last_knowledge_entry:{user_id}`).

**Q24. Что я сделал бы по-другому, если бы начинал заново?**
Два момента. Первое — сразу начал бы с тест-сета из реальных постов и evals по качеству summary, а не с кода. Это ускорило бы итерации промптов в 1.5-2 раза. Второе — не стал бы делать userbot: для продукта он тупиковый, лучше Bot API + веб-дашборд для показа дайджеста и тем.

**Q25. Почему именно в команду АССИСТ+?**
Цель курса — вернуть предпринимателю и автору час в день через AI-инструменты. TeleFlow решает ровно эту боль. Я хочу не стажироваться «где придётся», а расти там, где уже строят продукт в моей области.

- Нет — «не знаю». Есть — «в MVP это упрощено, в проде сделал бы так».
- Нет — «ну типа как-то так». Есть — «коротко: \[ответ]. Подробнее: \[причина]».
- Нет — спорить с вопросом. Есть — «хороший вопрос, давайте посмотрю».
