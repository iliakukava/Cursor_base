# TeleFlow — материалы для защиты итогового проекта

Документ объединяет: структуру питча, слайды, тайминг, промпт для генерации презентации ИИ и ответ на вопрос про хостинг на Hugging Face Spaces.

---

## 1. Что такое TeleFlow (ядро в одном абзаце)

**TeleFlow** — локальный Telegram userbot на Telethon: собирает **дайджест из непрочитанного** в выбранных папках, **размечает** посты через **OpenRouter (LLM)**, позволяет **сохранять** понравившиеся посты в **тематическую базу** (пересылка в ЛС → выбор темы → файлы `entry_*.json`) и **генерировать новый пост** по накопленным материалам (`/write <тема>`). Управление из личных сообщений с ботом, доступ по whitelist (`ALLOWED_USER_IDS`), есть режим **DRY_RUN** для безопасного просмотра без записи в канал/state. Демо-сценарий описан в `README.md`.

## 4. Живое демо (если разрешат экран)

По чеклисту из `README.md`:

1. Запуск `python -m teleflow --daemon`.
2. В ЛС userbot: `/digest`.
3. Переслать текстовый пост → выбрать тему.
4. `/write <тема>` — показать сгенерированный пост.

Одна цепочка убедительнее, чем много слайдов с кодом.

## 5. Заготовки под вопросы (5 минут Q&A)

- **Почему userbot, а не Bot API?** Нужен доступ к «своим» папкам и непрочитанному так же, как у пользователя Telegram.
- **Куда деваются данные?** Локально: `data/knowledge/`, `state.db`; секреты только в `.env`, не в репозитории.
- **Что если LLM ошибается?** В MVP человек финализирует текст; дальше — шаблоны, пост-обработка, другая модель.
- **Почему OpenRouter?** Единый HTTP API к разным моделям, смена модели через переменные окружения без переписывания ядра.

Итого 9–11 слайдов. На последнем слайде — короткий CTA: «вопросы».

## 7. Можно ли захостить TeleFlow на Hugging Face Spaces (в т.ч. с веб-интерфейсом)?

**Коротко: как основной хостинг для этого userbot — нет, это плохая идея.** Веб-URL Space (`*.hf.space`) не решает задачу «всегда онлайн Telegram-клиент».

**Почему:**

1. **Модель Spaces** — контейнер для веб-приложения (Gradio, Streamlit, Static HTML). TeleFlow — это **долгоживущий процесс** с Telethon-сессией, опросом Telegram, возможными интерактивными логинами; это не типичный «веб-дашборд», хотя теоретически можно запустить subprocess из Gradio (костыли и ограничения).
2. **Сон и перезапуски** на бесплатном тарифе Space **засыпает** без трафика → userbot отваливается от Telegram. Для бота нужен **постоянный** процесс (VPS, Railway с always-on, свой сервер, домашняя машина с systemd и т.д.).
3. **Сеть и политика** — исходящие соединения к Telegram с инфраструктуры HF могут быть нестабильны или меняться; для продакшена обычно берут VPS ближе к пользователю/прокси.
4. **Секреты** — API-ключи и сессия Telegram в Secrets Spaces возможны, но **сессия userbot** и **2FA** на практике удобнее заводить там, где у вас полный контроль (VPS/локально).

**Куда тогда сохранялась бы `data/` на Space?**

- Запись в каталог приложения (например `./data` в репозитории Space) идёт в **файловую систему контейнера**.
- На **обычном Space без отдельного постоянного тома** эта ФС **часто не гарантирует долговременное хранение** между пересборками/соном/миграцией — то есть **`state.db` и `data/knowledge/` могут пропасть** так же, как «временные» файлы в контейнере.
- Чтобы данные **жили между перезапусками**, нужен либо **явный persistent volume** (если доступен в вашем плане HF), либо вынос хранения во **внешнее** хранилище (S3, база, HF Dataset как файловое хранилище через API — всё это уже не «просто положил в папку проекта»).

**Итог:** TeleFlow задуман как **локальный или VPS-сценарий** с предсказуемой папкой `data/` на диске. HF Spaces с `@web` — скорее место для **демо-обёртки** (статус, ссылка на репо, инструкция), а не для боевого userbot и постоянной базы.
